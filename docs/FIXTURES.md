# Parity fixtures

Every `.docx` under `apps/demo/public/fixtures/` used for Word-parity work,
what feature cluster it exercises, its Word page count, and how to regenerate
it. Fixtures and their Word reference PDFs are **gitignored** — the committed
deliverable is the generator scripts listed here. Regenerate a fixture, export
its Word reference once, then run the pixel harness.

## How the bed works (recap)

- Fixtures: `apps/demo/public/fixtures/<name>.docx` (regenerated by scripts).
- Word ground truth: `parity/<name>-word.pdf` (exported once via local
  Microsoft Word; see "Exporting" below).
- Pixel comparison: `node scripts/parity-compare.mjs <names...> --base http://localhost:5299`
  writes `parity/out/<name>-p<N>.png` (Word | web | diff) and prints a
  per-page **structural %** and **raw %**. The dev server must be running
  (`npm run dev`, default port 5299).

The two numbers differ: **raw %** is the fraction of page pixels that differ
(stable, but dilutes on sparse pages and inflates on dense ones). **structural
%** blocks the page and aligns before scoring, so it is far more sensitive to
sub-line *vertical* offsets — a page whose content matches but sits one line
lower can score 90%+ structural at ~5% raw. Read them together: high structural
+ low raw = alignment drift; high both = real missing/wrong content.

## Validating BEFORE Word (required gate)

A fixture that Word has to REPAIR is not trustworthy ground truth — Word's
repaired in-memory doc may not match the `.docx` bytes our engine renders, and
a corrupt open can wedge the AppleEvents queue for the rest of an export batch.
So every fixture must pass validation before it goes to Word:

```
python3 scripts/validate-docx.py                   # all parity2-*.docx
python3 scripts/validate-docx.py <name> [<name>]   # specific fixtures
```

It checks zip integrity, XML well-formedness of every part, relationship /
content-type / r:id / footnote-endnote-id / numId cross-references, and — most
importantly — **OOXML child ORDER and duplicate singletons** in the ordered
complex types (`w:pPr`, `w:rPr`, `w:tcPr`, `w:tblPr`, `w:sectPr`, `w:trPr`,
`w:settings`). Out-of-order or duplicated children are the classic raw-XML
cause of Word's "corrupt / repaired" dialog, and both LibreOffice and our own
engine silently tolerate them — so this checker, not an external converter, is
the real Word-corruption detector. `scripts/export-parity2.sh` runs it as a
hard gate and refuses to export a fixture that fails.

A useful independent sanity check (strict enough to catch gross malformation,
lenient about ordering) is LibreOffice: `soffice --headless --convert-to pdf
--outdir /tmp/out -env:UserInstallation=file:///tmp/loprofile <fixtures...>`.

## Exporting Word references

```
scripts/export-parity2.sh <name> [<name> ...]      # this expansion's exporter (validates first)
```

It validates each fixture (hard gate — see above), then opens it in a
**running, unlocked** Microsoft Word via AppleScript `open file name` (the
LaunchServices `open -g` route stops delivering documents after a Word
force-quit), saves as PDF addressing the document **by name**, closes it
`saving no`, then verifies page count + first line. See operational notes at
the bottom.

### Sanitization gaps found in `scripts/sanitize-docx.py` (worked around here; not fixed)

While sanitizing the wild docs (and re-checking msa) three real leaks slipped
past the shared sanitizer; each was scrubbed post-hoc so the committed fixtures
are clean, but the script itself still has these blind spots:

1. **`scrub_rels` never runs.** In `main()` the `.rels` branch is gated by
   `if name.endswith(".xml")`, but `.rels` parts end in `.rels` — so external
   **hyperlink Targets are never redacted**. doe-rfp / wi-rfp / athabasca all
   shipped their real gov/edu hyperlink URLs until scrubbed by hand.
2. **`customXml/*` is untouched.** msa's Google-Docs e-signature part carried
   the real company name in a `SignerLabel`; wi-rfp's SharePoint part carried a
   `<Url>` doc-ID link. The sanitizer only rewrites `word/*` + `core.xml`.
3. **`HYPERLINK` field URLs and `docProps/app.xml` (`Company`, `Manager`, HLinks)
   survive.** `w:instrText` is deliberately skipped, and app.xml is not processed
   at all, so org names (`Vierbicher`, `NREL`, `Georgia Institute of Technology`,
   `Athabasca University`) and field-code URLs remained until scrubbed.

The scrub applied here preserves schema/namespace URIs (openxmlformats,
microsoft, w3.org, …) and only redacts content URLs → `https://example.com/`.

### Parity-harness incompatibility (leading empty span)

`scripts/parity-compare.mjs:80` waits for `.dxw-page span` in the default
**visible** state. `wild-athabasca` renders a leading empty paragraph whose
first span has width 0, so the wait times out and the run throws before scoring
— it also aborts a whole multi-fixture batch. Its baseline above was captured
with a throwaway local copy of the script using `{ state: "attached" }`; the
committed script is unchanged. Worth a one-word fix in the harness
(`state: "attached"`) so a leading empty span can't stall parity runs.

## Engine findings surfaced (documented, not fixed — `packages/` untouched)

No fixture crashed the engine and none produced a blank page; all 13 render and
all page counts match Word. Three genuine rendering gaps and several
vertical-metrics drifts were surfaced:

1. **Display equations (`m:oMathPara`)** render left-aligned at inline size, not
   centered as display math with display-mode operator/fraction sizing
   (`parity2-equations` p1; visible in `parity/out/parity2-equations-p1.png`).
2. **Cover-page decorative art** — the Facet cover's wpg freeform-geometry group
   with theme fills does not paint (text/SDTs render fine)
   (`parity2-coverpage` p1/p2).
3. **VML WordArt watermark + line numbering** — a `v:textpath` watermark in the
   header and `w:lnNumType` margin line numbers both render nothing; page
   borders in the same section are correct (`parity2-watermark` p1/p4).
4. **Drop-cap geometry** — framed drop-cap letter size/hang and body wrap-around
   differ from Word (`parity2-dropcap` p1/p2).
5. **Text-box wrap/position** — anchored wps text boxes and a rotated box are
   mis-placed and text wraps differently (`parity2-textboxes`).
6. **Vertical drift** — dense tables render with slightly shorter rows than Word
   and multi-section/landscape pages carry sub-line vertical offsets that
   accumulate; content and structure are correct (`parity2-nestedtables` p2,
   `parity2-sections` p2/p3/p5). These inflate structural% while raw% stays
   modest (~5-12%).

## Operational notes for exporting (hard-won)

- Word must be **running and the session unlocked** — AppleEvents that need a
  dialog time out (-1712) and LaunchServices `open -g` won't deliver documents
  into a locked session.
- After a Word **force-quit**, the `open -g` LaunchServices route silently stops
  delivering documents (`exists document` stays false with no dialog). Use the
  AppleScript `open file name` route (`scripts/export-parity2.sh` does this).
  It can pop a one-time Grant-File-Access sheet on a brand-new directory.
- Two failure modes, both now caught by `scripts/validate-docx.py` before Word:
  Word either **rejects** a file outright ("experienced an error opening" —
  throws on `open`, then a modal wedges later AppleEvents -1708/-1712) or opens
  it **with a silent repair** ("corrupt / recovered" dialog). A repaired open is
  just as bad as a rejection: its PDF reflects Word's fixed-up doc, not the
  bytes on disk. Defects hit while building this batch, all now guarded:
  - **`footnotePr`/`endnotePr` separators in settings.xml** (declaring footnote
    id -1/0) with **no** footnotes.xml/endnotes.xml part — Word errors with a
    FOOTNOTE-related message. This was in the shared settings of all 10
    hand-built fixtures in the first cut and triggered errors across the batch;
    the separators now live only in the notes fixture, which ships the parts.
  - **Empty table cell** (`<w:tc>` with only `<w:tcPr>`, no `<w:p>`) — every
    cell needs at least one block-level child. Word rejects the whole file.
  - **A cell ending in a nested `</w:tbl>`** — a cell's last block must be a
    paragraph; append a trailing `<w:p/>` after any nested table.
  - **Out-of-order `w:tcPr` children** — `gridSpan`/`vMerge` placed after
    `tcBorders`/`shd` (CT_TcPr is an ordered sequence: `tcW, gridSpan, vMerge,
    tcBorders, shd, textDirection, vAlign`). Repaired-open corruption in
    nestedtables — re-exported after the fix.
  - **Duplicate / out-of-order `w:tblBorders`** — a second `tblBorders` (or one
    after `tblCellMar`) in a `tblPr`. Repaired-open corruption in nestedtables.
  - **Out-of-order `w:pPr` children** — `jc` before `spacing` (spacing precedes
    jc in CT_PPr). Repaired-open corruption in textboxes — re-exported.
- Always **verify the exported PDF** (page count, first text line) — AppleEvents
  can report -1712/-1728 even when the save succeeded.
